# Test Failures due possible code bugs

## TeamNut.Tests.csproj - TeamNut.Repositories.UnitTests.MealPlanRepositoryTests.GeneratePersonalizedDailyMealPlan_ThreeMeals_SucceedsAndInsertsMealPlanMeals
- **Confidence**: High
- **Test File**: TeamNut.Tests\Repositories\MealPlanRepositoryTests.cs
- **Bug Location**: TeamNut\Repositories\MealPlanRepository.cs@166

### Analysis
The test inserts a UserData row with goal = NULL and expects the repository to fall back to the default goal_type value "general" when generating a meal plan. In GeneratePersonalizedDailyMealPlan the code assigns goal using udReader["goal"]?.ToString() ?? "general" (line ~166). That expression only uses the null-coalescing operator, so if udReader["goal"] is not null but contains DBNull.Value (or yields an empty string when ToString() is called), the left side is not null and the coalescing fallback is not used. Thus a database NULL (or an empty string representation) can result in goal being set to an empty string instead of "general". The test failure (goalType == "") demonstrates this. The fix is to treat DBNull.Value and empty strings as missing and explicitly fall back to "general". This is a production code bug in MealPlanRepository.GeneratePersonalizedDailyMealPlan.

### Suggested Fix
Replace the current assignment that reads the goal from the data reader so it explicitly checks for DBNull and empty strings and falls back to "general". For example, change the relevant part to:

```csharp
object goalObj = udReader["goal"];
goal = (goalObj != DBNull.Value && !string.IsNullOrWhiteSpace(goalObj?.ToString()))
       ? goalObj.ToString()
       : "general";
```

This ensures that when the database value is NULL or an empty string the repository will use the default "general" goal_type when inserting the MealPlan.
